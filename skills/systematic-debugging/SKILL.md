# 系统化调试

## 核心原则

**在尝试修复之前，务必先找到根本原因。只修症状就是失败。**

## 四个阶段

### 第一阶段：根因调查

在尝试任何修复之前：

1. **仔细阅读错误信息** — 不跳过错误或警告，完整阅读堆栈跟踪，记下行号、文件路径、错误码
2. **稳定复现** — 确认能否可靠触发，记录复现步骤；无法复现则收集更多数据
3. **检查近期变更** — git diff、最近提交、新依赖、配置变更、环境差异
4. **在多组件系统中收集证据** — 对每个组件边界记录进出数据，验证环境/配置传递，检查每一层状态
5. **跟踪数据流** — 错误值从哪里产生？谁用错误值调用了这里？持续向上追踪直到找到源头

### 第二阶段：模式分析

1. **找到可正常工作的示例** — 在同一代码库中找到类似的正常代码
2. **与参考实现对比** — 逐行阅读参考实现，在应用之前彻底理解该模式
3. **识别差异** — 列出正常代码和出问题代码之间的每一个差异，无论多小
4. **理解依赖关系** — 需要哪些其他组件？有哪些隐含假设？

### 第三阶段：假设与验证

1. **提出单一假设** — 清晰地陈述"我认为 X 是根本原因，因为 Y"，写下来
2. **最小化测试** — 最小改动验证假设，每次只改一个变量
3. **继续之前先验证** — 生效了？进入第四阶段。没生效？提出新假设
4. **当你不确定时** — 说"我不理解 X"，做更多调研

### 第四阶段：实施修复

1. **创建失败的测试用例** — 最简化的复现，修复前必须先有测试
2. **实施单一修复** — 修复已定位的根本原因，每次只改一处
3. **验证修复** — 测试通过了吗？其他测试没有被破坏吧？
4. **如果 3 次以上修复都失败** — 停下来质疑架构，与搭档讨论后再继续

## 红线——停下来，按流程走

- "先临时修一下，以后再排查" → 必须回第一阶段
- "试着改改 X 看看行不行" → 必须回第三阶段
- 一次性改多个地方 → 必须回第四阶段
- 跳过测试 → 必须回第四阶段
- 3 次以上修复失败 → 质疑架构，不要继续修
