# 逆向思维 — 从反面找突破点

> 从「如果失败了，最可能的原因是什么？」开始，反向推导风险和改进点。

## 适用场景

- 评估方案风险
- 识别盲点
- 做最坏情况准备

## 执行流程

1. **假设失败** — 跳转到项目失败/功能无人使用的场景
2. **倒推根因** — 列出所有可能导致失败的原因
   - 技术原因（性能、安全、兼容性）
   - 用户原因（学习成本、习惯、信任）
   - 业务原因（成本、时效性、竞品）
3. **正向验证** — 逐条确认「这个风险有多大？」
4. **反向改进** — 哪些根因可以在项目早期就消除？

## 核心问题清单

```
- 如果上线后没人用，最可能的原因是什么？
- 如果上线就崩溃，最可能踩到什么坑？
- 如果用户投诉，会投诉什么？
- 如果竞品先做了，我们的方案还有价值吗？
- 如果核心成员请假了，谁能接手？
```

## 输出

```markdown
## 逆向思维结果

### 失败场景
[描述最可能的失败场景]

### Top 3 风险
1. [风险] → [预防措施]
2. [风险] → [预防措施]
3. [风险] → [预防措施]

### 可消除的根因
- [在早期就能解决的问题]
```
