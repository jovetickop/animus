# /harness-code-debug

## 概述

系统化调试工作流。内置 4 阶段调试流程（根因调查→模式分析→假设验证→实施修复），结合 harness-code-plan 的任务规划编排框架。

**核心原则：** 不做根因调查，不许提修复方案。

## 双通道触发

| 触发方式 | 说明 |
|---------|------|
| **手动调用** | 在对话中输入 `/harness-code-debug` |
| **自动触发** | 检测到测试失败、构建失败、panic 时自动激活（严格匹配） |

自动触发时跳过启动检查，直接从**调试阶段**开始。
手动调用时走完整流程。

## 完整流程

### 阶段零：启动检查（仅手动调用）

1. 读取 `.claude/harness-cc/features.json`，检查任务状态：
   - 有 **pending/in_progress** 任务 → 使用 `AskUserQuestion` 询问继续完成还是丢弃
   - 有 **passed/completed** 任务 → 询问是否归档当前迭代
2. 根据用户选择执行对应操作（归档或继续）

### 阶段一：根因调查

在尝试任何修复之前，必须完成以下步骤：

**1. 仔细阅读错误信息**
- 不跳过错误或警告
- 完整阅读堆栈跟踪
- 记下行号、文件路径、错误码

**2. 稳定复现**
- 确认能否可靠触发
- 记录复现步骤
- 如果无法复现 → 收集更多数据，不猜测

**3. 检查近期变更**
- `git diff`、最近提交
- 新依赖、配置变更
- 环境差异

**4. 在多组件系统中收集证据**
- 对每个组件边界记录进出数据
- 验证环境/配置的传递
- 检查每一层的状态

**5. 跟踪数据流**
- 错误值从哪里产生？
- 谁用错误值调用了这里？
- 持续向上追踪直到找到源头

### 阶段二：模式分析

**1. 找到可正常工作的示例**
- 在代码库中找到类似的正常代码
- 对比参考实现

**2. 识别差异**
- 正常代码和出问题的代码有什么不同？
- 列出每一个差异，无论多小

**3. 理解依赖关系**
- 需要哪些其他组件？
- 有哪些隐含假设？

### 阶段三：假设与验证

**1. 提出单一假设**
- 写明："我认为 X 是根本原因，因为 Y"
- 要具体，不要含糊

**2. 最小化测试**
- 最小改动验证假设
- 每次只改一个变量

**3. 验证结果**
- 生效了？ → 进入阶段四
- 没生效？ → 提出新假设

### 阶段四：实施修复

**1. 先创建失败的测试用例**
- 最小复现
- 修复前必须先有测试

**2. 实施单一修复**
- 修复已定位的根本原因
- 每次只改一处

**3. 验证修复**
- 测试通过了吗？
- 其他测试没有被破坏吧？

**4. 如果 3 次以上修复都失败**
- 停下来质疑架构
- 与用户讨论后再继续

### 阶段五：任务规划

根因确定后，使用 feature-planner agent 将修复方案拆解为可执行任务。

**流程：**
1. 读取 `${CLAUDE_PLUGIN_ROOT}/agents/universal/feature-planner.md`
2. 将修复方案写入 `feature-detail.md`
3. 更新 `features.json` 创建修复任务（含依赖和验收标准）

### 阶段六：自动审查

修复任务完成后，自动触发代码审查：
- 通用检查（编码规范、错误处理、测试覆盖）
- 按项目类型的语言专项检查

### 阶段七：归档询问

审查通过后，使用 `AskUserQuestion` 询问用户是否归档当前迭代。

- 选"是" → 执行归档命令 `/harness-code-archive`
- 选"否" → 结束

## 红线规则

以下情况必须停下来：
- "先临时修一下" → 必须回阶段一
- "试着改改 X 看看行不行" → 必须回阶段三
- 一次改多个地方 → 必须回阶段四
- "大概没问题" → 必须有测试证据
- 3 次以上修复失败 → 质疑架构

## 辅助技术

详细调试技术参见 `${CLAUDE_PLUGIN_ROOT}/skills/systematic-debugging/`：

- **`root-cause-tracing.md`** — 沿调用栈反向追踪 bug，找到最初的触发点
- **`defense-in-depth.md`** — 找到根因后，在多个层级添加校验
- **`condition-based-waiting.md`** — 用条件轮询替代硬编码等待时间

## 验证标准

完成后应满足：
- [ ] 根因已明确记录
- [ ] 修复方案已拆分为 features.json 任务
- [ ] 失败的测试用例已创建
- [ ] 修复通过全部测试
- [ ] 代码审查通过
