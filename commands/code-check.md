# /code-check

把当前改动作为"当前任务的验收门"来审查。采用两层结构：**通用检查**（所有项目必做）+ **语言专项检查**（按 project-type 选择执行）。

## 前置步骤

1. 读取 `.claude/harness/project-config.json` 获取 `project-type`
2. 若文件不存在，默认按 `generic` 处理
3. 读取 `.claude/harness/features.json` 确认当前任务及其验收标准

## 第一层：通用检查（所有项目，始终执行）

### 1. 构建是否通过

- 执行项目的构建命令（从 CLAUDE.md 或 features.json 获取）
- 确认无编译错误、无链接错误
- 确认无未解决的外部依赖

### 2. 测试是否通过

- 执行项目的测试命令
- 确认所有测试用例通过（或标记为预期失败）
- 检查测试覆盖率是否达到项目要求（默认 80%+）

### 3. 代码质量

- **命名规范**：类名 `PascalCase`、函数 `PascalCase`、局部变量 `snake_case`、宏 `UPPER_SNAKE_CASE`
- **中文注释**：所有新增函数在声明和定义处必须有中文注释
- **错误处理**：关键路径是否有 try-catch 或错误返回；外部输入是否做校验
- **魔法数字**：是否有未命名的字面常量，应抽取为具名常量或枚举
- **函数长度**：单函数不超过 50 行，嵌套不超过 4 层
- **文件长度**：单文件不超过 800 行

### 4. Harness 状态一致性

- `features.json` 中的当前任务状态与进度日志一致
- `claude-progress.txt` 已记录最新操作摘要
- 验收标准已满足或明确记录差距

## 第二层：语言专项检查（按 project-type 选择执行）

### cpp-qt

| 检查项 | 说明 |
|--------|------|
| QObject 所有权 | 堆上创建的 QObject 是否明确了父对象或生命周期归属；是否保存了生命周期不清晰的裸指针 |
| MOC/UIC/RCC | 使用元对象的类是否声明 `Q_OBJECT`；`.ui` 和资源文件改动是否与代码同步；MOC/UIC/RCC 是否正常 |
| UI 线程阻塞 | 耗时逻辑是否移出 UI 线程（使用 QThread/QtConcurrent/异步）；槽函数中是否有阻塞调用 |
| 信号槽连接 | 是否优先使用新式函数指针语法；跨线程通信是否明确连接方式 |
| UI 布局与 sizePolicy | 主结构是否使用 Qt Layout；是否存在不必要的固定尺寸；文本输入/表格/日志区是否设置了合适的 sizePolicy |
| 用户可见文案 | 是否使用了 `tr()` 包裹用户可见字符串；文案是否中性明确 |

### cpp-cmake

| 检查项 | 说明 |
|--------|------|
| C++ 标准 | CMakeLists.txt 中是否显式声明了 C++ 标准（`CMAKE_CXX_STANDARD`） |
| 编译器警告 | 是否启用了合理的警告级别（MSVC: /W4, GCC/Clang: -Wall -Wextra） |
| 资源管理 | 是否使用了 RAII/智能指针，避免裸 new/delete |
| 构建配置 | CMake 是否使用 target_* 命令而非全局设置 |

### python

| 检查项 | 说明 |
|--------|------|
| 依赖一致性 | `pyproject.toml` / `requirements.txt` / `Pipfile` 中的依赖是否与实际 import 一致 |
| 类型注解 | 函数参数和返回值是否有类型注解；复杂数据结构是否使用 `TypedDict` / `dataclass` |
| pytest 覆盖 | 新增功能是否有 pytest 测试；配置了 pytest 覆盖报告时覆盖率是否达标 |
| 错误处理 | 外部输入和 API 调用是否有异常捕获；异常是否被吞没（bare except） |
| 异步使用 | 异步函数是否正确使用 `await`；事件循环管理是否合理 |

### node

| 检查项 | 说明 |
|--------|------|
| npm audit | `npm audit` 是否存在 high/critical 级别安全漏洞 |
| ESLint | 项目配置了 ESLint 时，是否通过 lint 检查 |
| 浏览器兼容 | 前端项目是否考虑了目标浏览器兼容性（browserslist、polyfill 等） |
| 依赖管理 | `package.json` 与 `package-lock.json` / `yarn.lock` 是否一致 |
| TypeScript | （如使用 TS）类型定义是否完整；`any` 使用是否合理 |

### rust

| 检查项 | 说明 |
|--------|------|
| clippy 警告 | `cargo clippy` 是否存在 warnings 或 errors |
| 所有权/借用 | 是否存在不必要的 clone；借用生命周期是否合理；`unsafe` 代码是否有安全说明 |
| 错误处理 | 是否合理使用 `Result` / `Option`；是否使用 `?` 操作符传播错误；自定义错误类型是否实现 `std::error::Error` |
| 测试覆盖 | 公有函数是否有单元测试或文档测试；`#[should_panic]` 是否有明确原因说明 |

### generic 和其他类型

无专项检查，仅执行通用检查。

## 输出格式

对每个发现的问题给出以下信息：

```
- 严重程度: high | medium | low
- 位置: 文件路径与行号（如可确定）
- 说明: 问题描述
- 影响: 该问题可能导致的影响
- 修复建议: 具体的修改建议
```

### 严重程度定义

| 级别 | 含义 |
|------|------|
| high | 构建失败、测试失败、安全漏洞、harness 状态不一致；必须修复后才能通过 |
| medium | 代码质量缺陷、违反项目规范但未导致失败；建议修复 |
| low | 风格和格式问题、文档不完整；可选修复 |

## 检查结论

审查结束时输出明确结论之一：

- **通过**：未发现 high 级别问题，所有验收标准已满足
- **有条件通过**：存在 medium/low 问题但不影响核心验收，记录待办项
- **不通过**：存在 high 级别问题，列出必须修复的事项

如果没有发现问题，也要明确说明，并补充剩余测试缺口或可改进项。
