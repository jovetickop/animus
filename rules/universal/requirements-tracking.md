# 需求追踪与差距分析规范

## 目的

确保每个开发周期开始时，已定义的需求与当前实现状态之间的差距被明确识别和记录。

## 触发时机

- 每次调用 `/animus-plan` 时自动执行
- 手动运行 `scripts/gap-analysis.py` 按需检查

## 差距分析输出格式

```text
| 功能 | 预期状态 | 实际状态 | 差距 | 原因 | 影响 |
|------|---------|---------|------|------|------|
| 用户登录 | 已实现 | ✅ passed | — | — | — |
| 密码重置 | 已定义 | ❌ pending | 未开始 | 依赖用户表 | 用户无法自助恢复 |
| 微信支付 | PRD 要求 | ❌ 未定义 | 未规划 | 第三方 SDK 待确认 | 影响上线时间 |
```

## 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| 功能 | ✅ | 功能名称或编号 |
| 预期状态 | ✅ | PRD/方案中定义的目标状态 |
| 实际状态 | ✅ | features.json 中的当前状态 |
| 差距 | ✅ | 已实现/未实现/部分实现/未定义 |
| 原因 | 未实现时必填 | 为什么未实现 |
| 影响 | 未实现时必填 | 如果不做的后果 |

## 验收标准

1. 差距分析表格必须包含所有已定义功能
2. 每个未实现项必须有原因和影响说明
3. 用户确认后才能进入实施阶段
4. 确认结果记录到 plan-context.md
