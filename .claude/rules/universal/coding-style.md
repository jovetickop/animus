# 通用编码规范

本规范用于约束所有项目中的日常代码修改，语言无关的通用规则。

## 1. 命名规则

| 类型     | 规则                       | 示例                   |
| -------- | -------------------------- | ---------------------- |
| 类型/类/接口 | `PascalCase` | `UserManager` |
| 函数/方法 | `camelCase` 或 `snake_case`（遵循项目既有风格） | `loadProject()` |
| 局部变量 | `snake_case` 或 `camelCase`（遵循项目既有风格） | `current_index` |
| 函数入参 | `snake_case` 或 `camelCase` | `file_path` |
| 常量/枚举 | `UPPER_SNAKE_CASE` | `MAX_RETRY_COUNT` |
| 文件名 | `snake_case` 或 `kebab-case`（遵循项目既有风格） | `user_manager` / `user-manager` |
| 私有成员 | 前导下划线（`_private`）或其他语言约定 | `_cache` |

## 2. 文件与目录组织

- 一个文件尽量只定义一个主要 public 类型。
- 模块/包内文件职责清晰，避免单文件承载多种不相关逻辑。
- 目录组织优先按功能/领域划分，而非按文件类型堆砌。

## 3. 文件结构

推荐顺序：

1. 文件头注释（如项目要求）
2. 引用保护 / `#pragma once`
3. 系统库 / 标准库引用
4. 第三方库引用
5. 项目内头文件引用
6. 前置声明
7. 常量 / 类型定义
8. 类声明

额外约束：

- 外部库使用 `<...>`，项目内头文件使用 `"..."`。
- 实现文件第一行通常是对应的头文件。
- 能前置声明的类型优先前置声明，避免头文件过度引用。

## 4. 引用顺序

原则：

- 同类引用放在一起。
- 被依赖程度更高、基础层更低的头文件靠前。
- 不要无序堆砌引用。

## 5. 格式要求

- 统一 4 空格缩进，禁止使用 Tab。
- 类、函数、`if/for/while/switch` 的左大括号换行。
- 单行 `if/for` 在逻辑非常简单时可省略大括号；多行分支必须加大括号。
- 每行建议不超过 100 个字符。
- 连续空行不超过 1 行。
- 不要为了"对齐好看"手动加入大量空格，格式统一交给格式化工具。

## 6. 注释规范

- 注释优先解释"为什么"，不要重复代码已经表达清楚的"做了什么"。
- 注释默认使用中文，保持简洁、明确。
- 公共接口、复杂状态机、线程边界、资源释放点建议补充注释。
- `TODO` 必须写清楚后续动作，不要只留"待优化"。

## 7. 核心设计约束

- 访问修饰符（`public`/`protected`/`private`）必须显式写出。
- 类职责保持单一，避免一个类同时承担 UI、业务、数据访问多重职责。
- 函数名必须体现动作或结果，避免 `DoWork`、`ProcessData` 这类空泛命名。
- 参数数量尽量控制在 4 个以内，过多时考虑封装参数对象。
- 避免深层嵌套；嵌套超过 4 层时优先提前返回、拆函数或抽策略。
- 避免带副作用的全局变量和静态初始化对象。
- 禁止魔法数字，改用常量、枚举或具名配置。
- 凡是不可变/只读的值，明确标记（语言对应的 const / final / readonly 等机制）。
- 优先使用 RAII 和智能指针管理资源。
- 裸指针只用于"不拥有对象"的场景，并保证生命周期清晰。

## 8. 提交前检查清单

- [ ] 命名符合规范
- [ ] 引用顺序清晰，无多余依赖
- [ ] 函数职责单一，复杂函数已拆分
- [ ] 无魔法数字
- [ ] 无深层嵌套（>4 层）
- [ ] 无调试残留、无无用注释代码
