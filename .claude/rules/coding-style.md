# C++/Qt 编码规范

本规范用于约束 C++/Qt 项目中的日常代码修改。

## 1. 命名规则

| 类型     | 规则                       | 示例                   |
| -------- | -------------------------- | ---------------------- |
| 类名     | `PascalCase`             | `MainWindow`         |
| 函数名   | `PascalCase`             | `LoadProject()`      |
| 成员变量 | 小驼峰                     | `projectPath`        |
| 局部变量 | `snake_case`             | `current_index`      |
| 函数入参 | `snake_case`             | `file_path`          |
| 文件名   | `snake_case`             | `main_window.cpp`    |
| 宏       | `UPPER_SNAKE_CASE`       | `DEFAULT_TIMEOUT_MS` |
| 枚举值   | `UPPER_SNAKE_CASE`       | `STATE_IDLE`         |
| 信号     | `Sig` 前缀               | `SigDataChanged`     |
| 槽函数   | `Slot` 前缀              | `SlotSaveClicked`    |
| 命名空间 | 小写或小写下划线           | `project_core`       |
| 模板参数 | 语义化名称，`snake_case` | `widget_type`        |

## 2. 文件与目录组织

- 一个头文件尽量只定义一个主要 public 类型。
- 头文件统一使用 `#pragma once`。
- 新建 `cpp/h/hpp` 文件统一使用 `UTF-8 BOM` 编码；若项目已有明确编码规范，以项目现有规范为准。
- 源文件和头文件命名要一一对应，例如 `main_window.h` / `main_window.cpp`。
- 目录组织优先按模块划分，不要把无关类堆在同一目录。
- Qt Widgets 工程优先保持 `src/`、`include/`、`ui/`、`resources/`、`tests/` 这样的清晰结构。

## 3. 头文件结构

推荐顺序：

1. 文件头注释（如项目要求）
2. `#pragma once`
3. 系统库 / 标准库引用
4. Qt 头文件引用
5. 项目内头文件引用
6. 前置声明
7. 枚举 / 常量 / 类型定义
8. 类声明

额外约束：

- 外部库使用 `<...>`，项目内头文件使用 `"..."`。
- `cpp` 文件中第一行通常是对应的头文件。
- 能前置声明的类型优先前置声明，避免头文件过度引用。
- 不要在头文件中包含明显只会在 `cpp` 中使用的重型依赖。

## 4. include 顺序

在 `cpp` 文件中推荐顺序如下：

1. 对应头文件
2. C++ 标准库
3. Qt 库
4. 第三方库
5. 项目内其他头文件

原则：

- 同类 include 放在一起。
- 被依赖程度更高、基础层更低的头文件靠前。
- 不要无序堆砌 include。

## 5. 格式要求

- 统一 4 空格缩进，禁止使用 Tab。
- 类、函数、`if/for/while/switch` 的左大括号换行。
- 单行 `if/for` 在逻辑非常简单时可省略大括号；多行分支必须加大括号。
- 每行建议不超过 100 个字符；若项目已有 `clang-format` 上限，以项目配置为准。
- 连续空行不超过 1 行。
- 不要为了“对齐好看”手动加入大量空格，格式统一交给 `clang-format`。

## 6. 注释规范

- 注释优先解释“为什么”，不要重复代码已经表达清楚的“做了什么”。
- 注释默认使用中文，保持简洁、明确。
- 公共类、关键接口、复杂状态机、线程边界、资源释放点建议补充注释。
- `TODO` 必须写清楚后续动作，不要只留“待优化”。

## 7. 核心设计约束

- `public`、`protected`、`private` 必须显式写出，成员变量统一放在 `private:`。
- 类职责保持单一，避免一个类同时承担 UI、业务、数据访问多重职责。
- 函数名必须体现动作或结果，避免 `DoWork`、`ProcessData` 这类空泛命名。
- 参数数量尽量控制在 4 个以内，过多时考虑封装参数对象。
- 避免深层嵌套；嵌套超过 4 层时优先提前返回、拆函数或抽策略。
- 避免带副作用的全局变量和静态初始化对象。
- 禁止魔法数字，改用常量、枚举或具名配置。
- 成员变量优先在构造函数初始化列表中初始化。
- 能用 `const` 的地方明确加 `const`。
- Qt 对象优先依赖父子关系管理生命周期。
- 非 Qt 对象优先使用 RAII 和智能指针管理资源。
- 裸指针只用于“不拥有对象”的场景，并保证生命周期清晰。

## 8. Qt 核心约束

- 继承 `QObject` 且使用元对象能力的类必须声明 `Q_OBJECT`。
- 优先使用新式信号槽连接语法。
- 槽函数中不要抛异常。
- 不要在非主线程直接操作 UI。
- 耗时逻辑不要阻塞 UI 线程；需要时使用线程、任务或异步机制。
- Qt 相关字符串、路径、容器、事件流程优先使用 Qt 原生 API，避免无必要混用。
- 修改 `.ui`、资源、对象名时，要同步检查 MOC、UIC、RCC 和代码引用。

## 9. 提交前检查清单

- [ ] 命名符合规范
- [ ] include 顺序清晰，无多余依赖
- [ ] 函数职责单一，复杂函数已拆分
- [ ] 无魔法数字
- [ ] 无深层嵌套（>4 层）
- [ ] 成员变量使用初始化列表
- [ ] QObject 生命周期明确
- [ ] 未在非主线程操作 UI
- [ ] 用户可见文案使用统一表达，并按需使用 `tr()`
- [ ] 无调试残留、无无用注释代码
