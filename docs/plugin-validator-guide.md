# 语义验证器 — AI 可读审查规则

> 本文件供 AI 助手在插件修改后阅读并执行语义质量审查。
> 与 `scripts/plugin-validator.py` 的 8 条确定性规则互补。
>
> 确定性规则已经检查通过的内容可以跳过。

---

## 如何使用

在以下场景请阅读本文件并逐条执行审查：

1. 新增/修改了 5 个以上的 Agent 或 Rule 文件后
2. 新增语言支持（创建完整的 `agents/{lang}/` 目录）后
3. 提交 PR 前作为自我审查清单
4. 代码审查时作为审查者的指导

---

## 规则 S1 — 描述准确性

每条 Agent 的 `description` frontmatter 必须在 1 句话内准确说明"AI 助手在什么场景下应该调用这个 Agent"。

| 等级 | 示例 |
|------|------|
| ✅ 正确 | "围绕当前 animus 任务实现最小闭环，适合处理单个 Txxx 任务的编码、文件修改、构建修复与状态同步。" |
| ⚠️ 模糊 | "C++ debugging"（太短，AI 不知道怎么触发） |
| ❌ 错误 | 直接复制了其他 Agent 的描述（导致路由冲突） |

**检查方法：** 对每个 Agent，用其 description 在项目内搜索——如果多个 Agent 匹配，说明描述不够独特。

---

## 规则 S2 — 职责不重叠

同一语言组内的 Agent 职责描述不应出现语义重叠。两个 Agent 都说"修复编译错误"时需要合并或明确分工。

**检查方法：** 按语言组读取 description，标记语义相似度 > 70% 的对。

---

## 规则 S3 — 规则不矛盾

同一语言组的 `rules/` 中，不应出现冲突的指导。例如一条说"使用 snake_case"，另一条说"使用 camelCase"。

**检查方法：** 按语言组读取 rules，标记明显的相反指令对。

---

## 规则 S4 — 降级路径声明

如果 Agent 或 Rule 在步骤中引用了外部工具（`clang-format`、`black`、`cargo fmt`），必须声明该工具不可用时的降级行为。

| 等级 | 示例 |
|------|------|
| ✅ 正确 | "Run clang-format。如果不可用则跳过格式化。" |
| ❌ 缺漏 | "Run cargo fmt"（无 fallback） |

**检查方法：** 扫描 Agent 文件中的 shell 命令引用，检查同一命令是否有对应的 hook 降级脚本。

---

## 规则 S5 — 激活步骤逻辑顺序

Agent 的激活步骤应按依赖顺序排列。后一步依赖前一步的输出时顺序不能颠倒。

| 正确顺序 | 错误顺序 |
|---------|---------|
| 1. 解析配置 | 1. 问候用户 |
| 2. 加载事实 | 2. 解析配置（语言可能不一致） |
| 3. 问候用户 | |

**检查方法：** 检查 Agent 定义中是否存在"先使用、后定义"的变量引用。

---

## 规则 S6 — 配置项存在声明

Agent 或 Rule 中引用的配置变量（如 `{communication_language}`）应在 config.toml 中有对应的默认值声明。

**检查方法：** 提取 Agent 文件中的所有 `{variable}` 引用，与 config.toml 的键集合对比。

---

## 规则 S7 — 文档同步

当 Agent/Rule 的行为变更时，对应的索引文档（agent-index.md）、路线图（bmad-optimization-roadmap.md）、架构文档（architecture.md）应同步更新。

**检查方法：**
1. 列出本次变更涉及的文件
2. 检查变更是否影响 Agent 的触发条件、输入输出或依赖关系
3. 如果是，确认索引文档已更新

---

## 规则 S8 — 国际化一致性

中文注释应与中文用户可见输出一致，避免中英混用（技术术语如 `git`、`CMake`、`features.json` 除外）。

**检查方法：** 扫描 Agent 文件中 > 20 字的可见文本段，检查是否同一种语言内混入了另一种语言的单词。
