# 优化任务 ⑤：`/animus-dev` 统一开发入口

> 对应路线图：Phase 1 — 快速见效 / P0（重编号后对应 ⑤）
> 解决：小改动走完整 7 问太重；bug fix 绕过 dev 流程没有任务跟踪

---

## 一、更改原因

### 1.1 当前问题

- `/animus-plan` 强制走 7 问 Grilling，bug fix 也走完整流程
- `/animus-debug` 是独立命令，修 bug 绕过 dev 流程，没有任务跟踪和审查门控
- 没有快速通道，用户改个按钮颜色也要 20 轮对话
- plan 和 dev 职责混在一起，用户需要知道用哪个命令
- 没有任务跟踪的门控，agent 可能直接改代码不记录

### 1.2 解决后的效果

- 一个入口 `/animus-dev` 处理所有场景：从改颜色到架构设计，含 bug 修复
- 五路自动分流：debug/oneshot/fast/light/full，按意图类型智能匹配
- AI 选路径后先输出「将使用 XX 路径」让用户确认（`autonomous` = true 时跳过）
- 所有路径都写 features.json，agent 没有任务不能写代码
- 支持 `--oneshot` / `--fast` / `--full` 参数手动强制路径

---

## 二、更改方案

### 2.1 路由设计

```
/animus-dev [--debug|--fast|--full] <意图>
```

**无参数时 AI 自动检测**，根据意图复杂度选路。

**自动恢复旧进度：** 启动时检测 `.claude/animus/memlog/` 是否有事件。有 → 自动 `rebuild` 恢复 features.json，输出「检测到上次进度，已恢复」。无 → 正常进入 dev 流程。

### 2.2 五路流程

```
                    ┌─────────────────────────────────────────┐
                    │  /animus-dev <意图>                      │
                    └──────────┬──────────────────────────────┘
                               │
                    ┌──────────▼──────────┐
                    │  features.json      │
                    │  有待办任务？        │
                    └──────┬──────┬───────┘
                     是    │     │   否
               ┌──────────▼┐  ┌─▼──────────────────────┐
               │ 直接实施   │  │ 先提问理解意图，自动分流          │
               │ implement │  └───┬────┬────┬──────┬──────┘
               │ → review  │ oneshot │fast │light│ full │
               └───────────┘  ┌─────▼────▼────▼──────▼─────┐
                              │  Grilling（按路径深度）       │
                              │  → 拆任务 → 用户确认          │
                              │  → 写入 features.json        │
                              │  → implement → review        │
                              └──────────────────────────────┘
```

### 2.3 五路 Grilling 深度

| | debug | oneshot | fast | light | full |
|--|-------|---------|------|-------|------|
| **触发条件** | bug 报告/异常/回归 | 零爆炸半径（改个颜色值） | 1-2 文件 / 小改动 | 3-10 文件 / 新增 | 架构级 / 跨模块 |
| **强制参数** | `--debug` | `--oneshot` | `--fast` | AI 自动 | `--full` |
| **提问数** | 3 问（调试专用） | 0 问 | 1 句确认 | 3 问 | 7 问 + 技法 |
| **提问内容** | 复现步骤/根因推断/修复策略 | "改什么？怎么改？" | "改什么？怎么改？" | "做什么？依赖什么？怎么验证？" | 验收/依赖/异常/性能/架构/风险/测试 |
| **spec 填充** | why + repro + root_cause + success | why + success，其余 AI 推断 | why + success，其余 AI 推断 | 5 字段较完整 | 5 字段详尽 |
| **任务条数** | 1 个 bugfix 任务 | 1 个轻量任务 | 1 个轻量任务 | 2-5 个任务 | 5+ 任务 |

**3 问明细（debug）：**

1. **复现步骤与影响范围** — 什么操作触发的？预期 vs 实际？
2. **根因初步推断** — 你怀疑是哪层的问题？有日志/堆栈吗？
3. **修复策略与副作用评估** — 修复会影响哪些已有功能？

**3 问明细（light）：**

1. 这个功能做什么的？解决了什么问题？
2. 依赖哪些现有模块/接口/数据？
3. 怎么验证做对了？验收步骤是什么？

**7 问明细（full）：**

1. **验收标准** — 从用户视角描述，完成后怎么验证
2. **前置依赖** — 已有模块、数据表、第三方服务
3. **异常流程** — 错误提示策略、回滚、降级
4. **性能/安全** — 加密、限流、超时、并发量级
5. **架构约束** — 分层、设计模式、技术栈限制
6. **风险** — 并发竞争、边界条件、第三方依赖
7. **测试策略** — 单元测试、集成测试、E2E

### 2.4 强制参数

| 参数 | 效果 | 适用场景 |
|------|------|---------|
| 无参数 | AI 自动检测意图类型和路径 | 大多数情况 |
| `--debug` | 强制 debug-path，3 问调试专用 | bug 修复
| `--oneshot` | 强制 oneshot-path，1 句确认后实施 | 眼见的 1 行修改，比如改一个颜色值 |
| `--fast` | 强制 fast-path，1 句确认 | 明确的小改动，比如改按钮颜色 |
| `--full` | 强制 full-path，走完整 7 问 | 复杂功能、架构变更 |

### 2.5 任务门控

implementer 指令中固化规则：

```
## 关键约束
- 你只能从 features.json 读取任务实施
- 如果没有 in_progress 任务，提示用户先运行 /animus-dev
- 不能根据"用户口头说"直接改代码
```

同时配合 PreToolUse hook（见写代码门控 Hook 任务）做硬拦截。

### 2.6 features.json 写入

所有路径都写 features.json：

```json
{
  "T004": {
    "title": "修按钮颜色",
    "status": "in_progress",
    "spec": {
      "why": "按钮和背景色太接近看不清",
      "capabilities": ["修改按钮文本颜色"],
      "constraints": ["只改 .qml 文件颜色值"],
      "non_goals": ["不修改按钮大小和位置"],
      "success": "按钮文字在白色背景下清晰可见"
    }
  }
}
```

### 2.7 合并原任务⑦（头脑风暴/产品发现）

full-path 的 7 问中集成 3 种技法：

| 用户说... | 自动启用技法 |
|-----------|------------|
| "我想做 X 但不知道从哪开始" | PRFAQ——先写假想发布公告 |
| "这个功能有没有更简单的方式" | 第一性原理——拆到基本元素再重建 |
| "这个功能还能怎么优化" | SCAMPER——七问检查法 |

### 2.8 改动文件

| 文件 | 改动 |
|------|------|
| 新建 `commands/animus-dev.md` | 主命令，五路路由 |
| 删除 `commands/animus-plan.md` | 直接移除，不留 deprecated |
| 删除 `commands/animus-debug.md` | 功能合并入 `/animus-dev` 的 debug-path |
| `plugin.json` | 注册新命令，删除旧注册 |
| `.claude/animus/config.toml` | 配置 `dev.default_path`、`dev.autonomous` |

### 2.9 与被替换命令的兼容

`/animus-plan` 和 `/animus-debug` 直接删除。功能由 `/animus-dev` 完全覆盖。

## 三、路径确认机制

AI 自动选路后，先输出「将使用 XX 路径」让用户确认：

```
检测到变更范围：按钮颜色值修改（1 个文件）
→ 推荐路径：oneshot（1 句确认后实施）
是否同意？(Y/n) > 
```

用户确认后继续。`config.toml` 中 `dev.autonomous = true` 时跳过确认，AI 全权决策。

## 四、架构影响评估

| 维度 | 评估 |
|------|------|
| 性能 | 无显著影响——oneshot 路径甚至更轻量 |
| 兼容性 | `/animus-plan` 和 `/animus-debug` 直接删除 |
| 降级 | AI 误判 bug vs feature 时，用户可用 `--full` 或 `--debug` 手动覆盖 |

## 五、验证方法

1. `/animus-dev PDF 导出崩溃` → 确认 debug-path，3 问调试专用 Grilling
2. `/animus-dev 修按钮颜色` → 确认 fast-path，1 问后创建轻量任务
3. `/animus-dev --full 加 PDF 导出` → 确认走完整 7 问
4. `/animus-dev --oneshot 改个颜色` → 确认1 句确认后实施
5. 确认 debug-path 创建的 features.json 有 type/severity/repro_steps/root_cause
6. 确认 implementer 在没有任务时拒绝写代码
7. 确认 `/animus-plan` 和 `/animus-debug` 已移除
